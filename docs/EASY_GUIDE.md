# クリーンアーキテクチャ + DDD をめちゃくちゃ簡単に説明する

---

## クリーンアーキテクチャって何？

一言で言うと「**役割分担をきっちり決めたコードの書き方**」です。

---

## 図書館で例えてみよう

図書館には色んな人が働いています。

```
来館者（ブラウザ）
    ↓
受付スタッフ        ← コントローラー
    ↓
司書（手順を知っている人）  ← ユースケース
    ↓
本そのもの（ルールを持っている）  ← エンティティ・値オブジェクト
    ↓
書庫担当（本の保管・取り出し）  ← リポジトリ
    ↓
書庫（データベース）
```

### 受付スタッフ（コントローラー）の仕事
- 来館者の話を聞く
- 司書に伝える
- 結果を来館者に伝える
- **本の中身や書庫の場所は知らなくていい**

### 司書（ユースケース）の仕事
- 「本を登録する手順」を知っている
- 「同じ本がすでにあるか確認する」などの段取りをする
- **受付の仕事も、書庫の場所も知らなくていい**

### 本そのもの（エンティティ・値オブジェクト）
- 「貸し出し中の本はさらに貸せない」というルールを持っている
- 「ISBNは13桁でないといけない」というルールを持っている
- **図書館の受付も書庫も知らなくていい**

### 書庫担当（リポジトリ）の仕事
- 本を棚にしまう
- 本を棚から取り出す
- **本の貸し出しルールは知らなくていい**

---

## DDDって何？

DDD（ドメイン駆動設計）は「**現実のビジネスのルールをそのままコードで表現しよう**」という考え方です。

### 「ドメイン」って何？
ドメイン = そのシステムが扱う「業務の世界」のことです。

- 図書館システムなら → 本・貸し出し・返却・会員
- 自動点呼システムなら → 運転者・点呼・アルコール検査・合否判定

### なぜドメインを大事にするの？

普通のコードはこう考えます。
```
「まずDBのテーブルを作ろう」
「次にそれを取得するコードを書こう」
```

DDDはこう考えます。
```
「まず業務のルールを整理しよう」
「貸し出し中の本は貸せない、というルールをコードで表現しよう」
「DBはあとで考えよう」
```

業務のルールが変わっても、DBが変わっても、それぞれ影響が少なくなります。

---

## 3つの重要な概念

### 値オブジェクト（Value Object）
「**正しい値しか生まれてこられない入れ物**」

例：ISBNという値オブジェクト
- `new ISBN("9784123456789")` → OK（13桁）
- `new ISBN("abc")` → 即エラー（数字じゃない）

ISBNオブジェクトが存在しているということは、それは正しいISBNです。
バリデーションのルールが1か所にまとまっています。

### エンティティ（Entity）
「**IDを持ち、ビジネスのルールを知っているオブジェクト**」

例：本（Book）というエンティティ
- `$book->lend()` → 貸し出す（貸し出し中ならエラー）
- `$book->return()` → 返却する（返却済みならエラー）

「貸し出し中の本はさらに貸せない」というルールが本自身の中にあります。

### リポジトリ（Repository）
「**保存・取得の窓口**」

例：本のリポジトリ
- `$repository->save($book)` → 本を保存する
- `$repository->findByISBN($isbn)` → ISBNで本を探す

DBの種類（MySQL・PostgreSQL）を変えても、他のコードは影響を受けません。

## なぜ分けると良いの？

### 変更に強くなる

```
「ISBNのルールが変わった！」
  → 値オブジェクト（ISBN.php）だけ直せばOK

「DBをMySQLからMongoDBに変えたい！」
  → リポジトリの実装だけ変えればOK

「登録の手順が変わった！」
  → ユースケース（RegisterBookUseCase.php）だけ直せばOK
```

### テストが書きやすくなる

```
「本の貸し出しルールをテストしたい」
  → Book.phpだけテストすればOK
  → DBもHTTPも必要ない
```

### チームで開発しやすくなる

```
Aさん → コントローラーを担当
Bさん → ビジネスロジックを担当
Cさん → DB周りを担当
```

それぞれが独立しているので、お互いの作業が邪魔になりにくいです。

---

## 依存の向きって何？

このプロジェクトで一番大事なルールです。

```
コントローラー → ユースケース → ドメイン ← リポジトリ実装
```

矢印は「依存している方向」です。

- コントローラーはユースケースを知っている
- ユースケースはドメインを知っている
- **ドメインは誰も知らない（依存していない）**
- リポジトリ実装はドメインのインターフェースを知っている

ドメイン（ビジネスのルール）が一番大事なので、誰にも依存させません。
これが「クリーン」の意味です。
